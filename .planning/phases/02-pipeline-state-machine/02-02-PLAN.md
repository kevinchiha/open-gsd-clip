Implement the DAG dependency resolver that determines which pipeline phases can run in parallel vs must run sequentially.

Purpose: The resolver is the intelligence behind parallel execution. Without it, all phases run sequentially. With it, independent phases run concurrently, reducing total pipeline time proportional to the parallelism in the dependency graph.

Output: A single resolver module with comprehensive tests covering normal cases, edge cases, and error conditions.
</objective>

**ResolverError type:**
```typescript
type ResolverError =
  | { type: 'cyclic_dependency'; involvedPhases: number[] }
  | { type: 'missing_dependency'; phase: number; missingDep: number };
```

**buildExecutionPlan function:**
`function buildExecutionPlan(phases: PhaseInput[]): Result<ExecutionPlan, ResolverError>`

Implementation using Kahn's algorithm with level extraction per RESEARCH.md Pattern 3:

1. **Normalize inputs:** Treat undefined/null dependsOn as empty array.
2. **Validate references:** For every dependency, check it exists in the input set. Return `{ type: 'missing_dependency' }` error on first missing ref found.
3. **Build adjacency structures:** in-degree map and forward adjacency list (phase -> phases that depend on it).
4. **Initialize queue:** All phases with in-degree 0 form the first level.
5. **Level-by-level BFS:** Process each level as a parallel group. For each processed phase, decrement in-degrees of its dependents. Dependents reaching in-degree 0 join the next level.
6. **Cycle detection:** After BFS completes, if processed count != total phases, a cycle exists. Report the unprocessed phases in the error.
7. **Sort phase numbers within each group** for deterministic output (tests depend on this).

Handle edge cases per user decisions:
- Empty input: return { ok: true, value: { groups: [], phaseOrder: [] } }
- No dependencies on any phase: all phases in one group
- Self-referencing dependency (phase depends on itself): caught by cycle detection
