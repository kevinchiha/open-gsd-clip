# Phase 06.1: Paperclip Plugin Packaging and Integration Readiness - Research

**Researched:** 2026-03-18
**Domain:** npm package structure, tsup build configuration, Paperclip plugin SDK worker protocol
**Confidence:** HIGH

## Summary

This phase addresses three critical blockers preventing the plugin from being installed and loaded by Paperclip: (1) no `worker.ts` entry point, (2) no `main`/`exports`/`paperclipPlugin` fields in `package.json`, and (3) no build configuration or `dist/` output.

The most significant finding is that `@paperclipai/plugin-sdk` has been **published to npm** (v2026.318.0, published today) since Phase 1 research. This SDK provides `definePlugin()`, `runWorker()`, and `startWorkerRpcHost()` -- the canonical way to create a plugin worker. The SDK also exports typed JSON-RPC protocol helpers, `PluginContext` types, and a test harness. However, adopting the SDK would require significant refactoring of the existing hand-rolled RPC layer. The pragmatic path for this phase is to create a thin `worker.ts` that wraps the existing `startPlugin()` function, matching what Paperclip's worker manager expects, without rewriting the RPC internals.

The second critical finding is how Paperclip's `plugin-loader.ts` discovers and spawns plugins. It reads `package.json`, looks for the `paperclipPlugin.manifest` key to find the manifest module, and the `entrypoints.worker` field in the loaded manifest to find the worker entry point. The host uses `node:child_process.fork()` (not `spawn()`) with `stdio: ["pipe", "pipe", "pipe", "ipc"]`. The `initialize` RPC now expects a response of `{ ok: true, supportedMethods?: string[] }` -- NOT the manifest (which is what our current handler returns). This is a protocol mismatch that must be fixed.

**Primary recommendation:** Create `src/plugin/worker.ts` as the Paperclip entry point, create `src/plugin/manifest.ts` as a standalone manifest module, create `tsup.config.ts` with proper entry points, add `paperclipPlugin` key to `package.json`, and fix the `initialize` RPC response. Do NOT adopt the full plugin SDK in this phase -- that is a future refactor.

## Architecture Patterns

### Required Package Structure (Paperclip Plugin Contract)
```
dist/
  manifest.js          # export default manifest (loaded by plugin-loader)
  plugin/
    worker.js          # entry point forked by Paperclip (default export + auto-start)
    index.js           # existing startPlugin() and exports
    rpc-handler.js     # existing RPC routing
    rpc-transport.js   # existing stdin/stdout JSON transport
  agents/              # all other source modules bundled
  api/
  bridge/
  notifications/
  orchestrator/
  pipeline/
  shared/
  signals/
```

### Pattern 1: Standalone Worker Entry Point
**What:** A thin `worker.ts` file that imports `startPlugin()` and calls it when run as a child process. This is the file Paperclip's `fork()` targets.
**When to use:** Always -- Paperclip requires this exact contract.
**Example:**
```typescript
// src/plugin/worker.ts
// Paperclip forks this file as a child process.
// It auto-starts the plugin when loaded.
import { startPlugin } from './index.js';

startPlugin();
```

### Pattern 2: Separate Manifest Module
**What:** The manifest is exported as `default` from a standalone module. Paperclip's plugin-loader imports this module via dynamic `import()` to read the manifest before spawning the worker.
**When to use:** Required by Paperclip plugin-loader.
**Example:**
```typescript
// src/plugin/manifest.ts
import type { PaperclipPluginManifestV1 } from './types.js';

const manifest: PaperclipPluginManifestV1 = {
  id: '@open-gsd/clip',
  apiVersion: 1,
  version: '0.1.0',
  displayName: 'GSD Orchestrator',
  description: 'Automates the full GSD development pipeline via agent orchestration',
  categories: ['automation'],
  capabilities: [
    'data.read',
    'data.write',
    'plugin.state',
    'events.subscribe',
    'agent.tools.register',
  ],
  entrypoints: {
    worker: './dist/plugin/worker.js',
  },
};

export default manifest;
export { manifest };
```

### Pattern 3: paperclipPlugin Key in package.json
**What:** The `paperclipPlugin` key in `package.json` tells Paperclip's plugin-loader where to find the manifest and worker modules.
**When to use:** Required for plugin discovery and installation.
**Example:**
```json
{
  "paperclipPlugin": {
    "manifest": "./dist/manifest.js",
    "worker": "./dist/plugin/worker.js"
  }
}
```

### Pattern 4: Initialize RPC Response Fix
**What:** Paperclip's worker manager expects `initialize` to return `{ ok: true, supportedMethods?: string[] }`. Our current handler returns the manifest object.
**When to use:** Must fix in rpc-handler.ts.
**Example:**
```typescript
// In buildMethods():
async initialize(params, id) {
  // params contains: { manifest, config, instanceInfo, apiVersion }
  // from the host -- we can use these to configure the plugin

  log.info('Plugin initialized');

  return success({
    ok: true,
    supportedMethods: [
      'initialize',
      'health',
      'shutdown',
      'onEvent',
      'executeAction',
    ],
  }, id);
}
```

### Anti-Patterns to Avoid
- **Bundling node_modules into dist/:** tsup should externalize all `dependencies` and `@paperclipai/*` packages. Paperclip resolves these from the plugin's own `node_modules/`.
- **Using `spawn()` in integration tests when Paperclip uses `fork()`:** The test harness uses `spawn('node', ['--import', 'tsx', ...])` which is fine for dev testing, but be aware that Paperclip uses `fork()` which adds an IPC channel. The stdio behavior is the same.
- **Building CJS output:** The package is `"type": "module"` and all imports use `.js` extensions. Build ESM only (`format: ['esm']`). Paperclip's worker manager uses `fork()` which respects the module type.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| tsup entry point configuration | Manual esbuild/rollup config | `tsup.config.ts` with named entry points | tsup handles ESM output, source maps, declaration files, and external detection |
| Package exports map | Manual `main`/`module` fields | `exports` field with conditional exports | Node.js 22 uses `exports` as the primary resolution mechanism |
| Worker auto-start detection | Complex `import.meta.url` comparison | Simple `startPlugin()` call at module scope in worker.ts | worker.ts IS the entry point -- no detection needed, it always auto-starts |

**Key insight:** The existing `isMainModule` detection in `index.ts` (checking `process.argv[1]`) is for dev mode (`npm run dev`). The production worker.ts should unconditionally start because Paperclip only forks it when it intends to run it.

## Common Pitfalls

### Pitfall 1: tsup Default Entry Point
**What goes wrong:** Running `tsup` with no config or arguments uses `src/index.ts` as the only entry point. This produces a single `dist/index.js` -- no `dist/plugin/worker.js` or `dist/manifest.js`.
**Why it happens:** tsup's default is "just bundle src/index.ts". The project had no tsup config file.
**How to avoid:** Create `tsup.config.ts` with explicit entry points for manifest, worker, and the main barrel export.
**Warning signs:** `dist/` only contains `index.js` and `index.d.ts`.

### Pitfall 2: Initialize Response Mismatch
**What goes wrong:** Paperclip's worker manager checks `result.ok` from the initialize response. If it is not truthy, the worker is killed with "Worker initialize returned ok=false".
**Why it happens:** Our current `initialize` handler returns `success(manifest, id)` which makes `result` equal to the manifest object. The manifest has no `ok` property, so `result.ok` is `undefined` (falsy).
**How to avoid:** Change the initialize handler to return `{ ok: true, supportedMethods: [...] }`.
**Warning signs:** Paperclip logs "Worker initialize returned ok=false" and kills the worker immediately after fork.

### Pitfall 3: fork() vs spawn() IPC Channel
**What goes wrong:** `fork()` creates an IPC channel (the 4th stdio descriptor). If the worker writes to the IPC channel or tries to use `process.send()`, unexpected behavior occurs.
**Why it happens:** `fork()` adds `"ipc"` to the stdio array. Our code only uses stdin/stdout for JSON-RPC, which is correct, but developers may accidentally use `process.send()` thinking it is available.
**How to avoid:** Never use `process.send()` in the plugin worker. All communication goes through stdin/stdout JSON-RPC lines.
**Warning signs:** "channel closed" errors or orphaned IPC messages.

### Pitfall 4: External Dependencies Not Marked
**What goes wrong:** tsup bundles `zod`, `pino`, `execa`, `@paperclipai/shared` etc. into the dist output. This bloats the bundle, causes version conflicts, and may break runtime resolution.
**Why it happens:** By default tsup bundles everything except node built-ins.
**How to avoid:** Set `external` in tsup config to externalize all `dependencies` and `@paperclipai/*` packages. Better: use `noExternal: []` combined with tsup's automatic externalization of `dependencies`.
**Warning signs:** Dist files are > 1MB, or runtime errors about duplicate module instances.

### Pitfall 5: Source Maps and Declaration Files
**What goes wrong:** No `.d.ts` files means TypeScript consumers cannot import types. No source maps means stack traces point to bundled code.
**Why it happens:** tsup does not emit declarations by default.
**How to avoid:** Enable `dts: true` and `sourcemap: true` in tsup config.
**Warning signs:** Import errors in consuming code, unhelpful stack traces.

### Pitfall 6: Manifest entrypoints.worker Path
**What goes wrong:** The manifest declares `entrypoints.worker: './dist/plugin/worker.js'`. Paperclip's `resolveWorkerEntrypoint()` resolves this relative to the package root. If the actual output file is at a different path (e.g., `dist/worker.js` instead of `dist/plugin/worker.js`), the worker is not found.
**Why it happens:** tsup flattens entry point paths by default unless `bundle: false` or entry points use explicit output mapping.
**How to avoid:** Verify the tsup output structure matches the manifest path. Use tsup's `entry` map syntax: `{ 'plugin/worker': 'src/plugin/worker.ts' }`.
**Warning signs:** "Worker entrypoint not found" error from Paperclip plugin-loader.

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Hand-rolled JSON-RPC (our implementation) | `@paperclipai/plugin-sdk` with `definePlugin()` + `runWorker()` | 2026-03-18 (SDK published today) | SDK handles all protocol details, provides typed PluginContext. Our hand-rolled approach still works but is now "legacy" |
| Manifest embedded in rpc-handler.ts | Manifest as separate default-exported module at `dist/manifest.js` | Current Paperclip spec | Paperclip loads manifest before spawning worker; must be a separate importable module |
| `initialize` returns manifest | `initialize` returns `{ ok: true, supportedMethods: [...] }` | Current Paperclip worker-manager | Protocol change from original PLUGIN_SPEC; manifest is loaded separately by the loader |

**Deprecated/outdated:**
- Returning manifest from `initialize` RPC -- Paperclip now loads the manifest via the loader before spawning the worker. The initialize RPC receives the manifest as INPUT (in params) and returns a readiness signal.
- Phase 1 research note "Plugin SDK not published to npm" -- it is now published as `@paperclipai/plugin-sdk@2026.318.0`.

## Open Questions

1. **@paperclipai/shared version mismatch**
   - What we know: We have `@paperclipai/shared@0.3.1` installed. Latest is `2026.318.0`. The plugin-sdk depends on `@paperclipai/shared@2026.318.0`.
   - What's unclear: Whether the `PaperclipPluginManifestV1` type has changed between 0.3.1 and 2026.318.0 in ways that affect our manifest.
   - Recommendation: Do NOT upgrade `@paperclipai/shared` in this phase. The manifest interface is stable (verified by reading the plugin-sdk types which re-export it). Upgrading shared types is a separate concern and risks breaking the 491 passing tests. If Paperclip's loader rejects our manifest at runtime, upgrade then.

2. **Full SDK adoption timeline**
   - What we know: The SDK provides `definePlugin()`, `runWorker()`, `startWorkerRpcHost()`, and a full `PluginContext` implementation that handles JSON-RPC automatically.
   - What's unclear: Whether adopting the SDK would require restructuring the PipelineRunner, agent spawning, and notification services that currently use our custom HostServices interface.
   - Recommendation: Defer SDK adoption to a future phase. The current hand-rolled JSON-RPC works correctly and the SDK's `PluginContext` client interfaces overlap but don't align perfectly with our `HostServices` interface pattern.

3. **Build output at npm publish time vs install time**
   - What we know: `dist/` is gitignored. Paperclip installs plugins via `npm install` which runs the npm lifecycle. The `prepare` script can trigger builds.
   - What's unclear: Whether Paperclip expects pre-built packages (with dist/ already in the tarball) or runs `npm run build` after install.
   - Recommendation: Add `"files": ["dist"]` to package.json AND add `"prepublishOnly": "npm run build"` script. This ensures the published npm tarball includes built files. Also add a `"prepare": "npm run build"` script for local `npm link` workflows.
