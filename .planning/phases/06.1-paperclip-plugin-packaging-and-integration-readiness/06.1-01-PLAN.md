  <files>src/plugin/manifest.ts, src/plugin/worker.ts, src/plugin/rpc-handler.ts, src/plugin/index.ts</files>
  <action>
1. Create `src/plugin/manifest.ts`:
   - Import `PaperclipPluginManifestV1` type from `./types.js`
   - Define the manifest object (move from rpc-handler.ts) with:
     - id: '@open-gsd/clip'
     - apiVersion: 1
     - version: '0.1.0'
     - displayName: 'GSD Orchestrator'
     - description: 'Automates the full GSD development pipeline via agent orchestration'
     - categories: ['automation']
     - capabilities: ['data.read', 'data.write', 'plugin.state', 'events.subscribe', 'agent.tools.register']
     - entrypoints: { worker: './dist/plugin/worker.js' }
   - Export as both `default` and named `{ manifest }` (Paperclip's loader uses `mod.default ?? mod`)

2. Create `src/plugin/worker.ts`:
   - Import `startPlugin` from `./index.js`
   - Call `startPlugin()` unconditionally at module scope (no isMainModule check -- this file is ONLY loaded when Paperclip forks it)
   - Add JSDoc comment explaining this is the Paperclip fork target

3. Update `src/plugin/rpc-handler.ts`:
   - Remove the inline `manifest` object definition (lines 18-36)
   - Add `import { manifest } from './manifest.js';` at the top
   - Keep the `export { manifest }` at the bottom (backward compat for index.ts re-export)
   - Change the `initialize` handler to return `{ ok: true, supportedMethods: ['initialize', 'health', 'shutdown', 'onEvent', 'executeAction'] }` instead of `success(manifest, id)`. This matches Paperclip's plugin-worker-manager.ts which checks `result.ok` truthy.

4. No changes needed to `src/plugin/index.ts` -- it already imports `{ manifest }` from `./rpc-handler.js` which re-exports from the new manifest module.
  </action>
