  <name>Task 2: MergeQueue -- ordered merge with skip-on-failure</name>
  <files>src/orchestrator/merge-queue.ts, src/orchestrator/merge-queue.test.ts</files>
  <behavior>
    - enqueue(phaseNumber): marks phase completed, triggers drain in roadmap order
    - drain merges phases in mergeOrder sequence: if phase 3 finishes before phase 1, phase 3 waits
    - markFailed(phaseNumber): skips phase in merge order, triggers drain to unblock later phases
    - isComplete(): returns true when all phases merged or skipped
    - Merge callback receives phaseNumber and is awaited (async)
  </behavior>
  <action>
    Write tests first (RED) then implement (GREEN).

    **Test approach:** Use a mock onMerge callback (vi.fn().mockResolvedValue(undefined)) to track merge order. Tests verify:
    1. Phases enqueued in order merge immediately in sequence
    2. Out-of-order completion: enqueue(3) then enqueue(1) then enqueue(2) -- onMerge called in order [1, 2, 3]
    3. markFailed(2) then enqueue(1) then enqueue(3) -- onMerge called for [1, 3], skipping 2
    4. isComplete returns false until all phases merged/skipped, true after
    5. Empty mergeOrder: isComplete returns true immediately
    6. Single phase: enqueue works correctly, isComplete becomes true
    7. markFailed for phase not in mergeOrder is a no-op (no throw)

    **Implementation (src/orchestrator/merge-queue.ts):**

    Follow research Pattern 2 with the skip-on-failure extension from the Code Examples section.

    - Export `MergeQueue` class
    - Constructor: `(mergeOrder: number[], onMerge: (phaseNumber: number) => Promise<void>)`
    - Private state: `completed: Set<number>`, `skipped: Set<number>`, `nextToMerge: number` (first in mergeOrder), `mergeOrder: number[]`
    - enqueue(phaseNumber): add to completed set, call drain()
    - markFailed(phaseNumber): add to skipped set, call drain() to unblock waiting merges
    - drain(): iterate mergeOrder from nextToMerge position. For each phase: if skipped, advance pointer and continue. If completed, call onMerge, remove from completed, advance pointer. If neither, break (not ready).
    - isComplete(): nextToMerge === -1 (sentinel for "past end of mergeOrder")
    - Use `createChildLogger('merge-queue')` for structured logging

    Handle edge case: empty mergeOrder sets nextToMerge to -1 immediately (isComplete = true from start).
