1. **src/bridge/types.ts**: Domain types matching gsd-tools.cjs output (camelCase, converted from snake_case JSON):
   - `RoadmapPhase`: number, name, goal, dependsOn, planCount, summaryCount, hasContext, hasResearch, diskStatus, roadmapComplete
   - `RoadmapAnalysis`: milestones, phases, phaseCount, completedPhases, totalPlans, totalSummaries, progressPercent, currentPhase, nextPhase, missingPhaseDetails
   - `PhaseDefinition`: found, phaseNumber, phaseName, goal, successCriteria, section
   - `ProjectState`: gsdStateVersion, milestone, milestoneName, status, stoppedAt, lastUpdated, lastActivity, progress (totalPhases, completedPhases, totalPlans, completedPlans, percent)
   - `PhasePath`: found, directory, phaseNumber, phaseName, plans, summaries
   - `BridgeOptions`: toolsPath (optional override), timeout (default 30000)

2. **src/bridge/errors.ts**: Typed error hierarchy per CONTEXT.md:
   - `GsdBridgeError extends GsdOperationalError` with `command` property
   - `GsdToolsNotFoundError extends GsdBridgeError` with `searchedPaths: string[]` property
   - `GsdParseError extends GsdBridgeError` with `rawOutput: string` property
   - `GsdTimeoutError extends GsdBridgeError` with `timeoutMs: number` property
   - Import `GsdOperationalError` from `../shared/errors.js`

3. **src/bridge/schemas.ts**: Zod 3 schemas matching gsd-tools.cjs JSON output (snake_case keys -- transformation to camelCase happens in bridge/index.ts):
   - `RoadmapPhaseSchema`, `RoadmapAnalysisSchema`, `PhaseDefinitionSchema`, `StateJsonSchema`, `FindPhaseSchema`
   - Use the exact field names from the research file's verified output structures
   - Use `z.string()` for numeric strings that gsd-tools returns as strings (phase numbers, progress values)

4. **src/bridge/discovery.ts**: Auto-discovery of gsd-tools.cjs path.
   - `discoverGsdToolsPath(): string` (synchronous -- file existence checks are fast)
   - Step 1: Check `process.env.GSD_TOOLS_PATH` -- if set, verify file exists with `fs.existsSync`, return if found
   - Step 2: Check `path.join(os.homedir(), '.claude/get-shit-done/bin/gsd-tools.cjs')` -- return if found
   - Step 3: Try `import.meta.resolve('get-shit-done-cc')` to find package, derive path -- return if found
   - Step 4: Throw `GsdToolsNotFoundError` with all three searched paths listed
   - Use shared logger for debug-level logging of discovery steps

5. **src/bridge/discovery.test.ts**: Unit tests for discovery logic.
   - Test: returns GSD_TOOLS_PATH env var value when set and file exists
   - Test: falls back to default path when env var not set
   - Test: throws GsdToolsNotFoundError when nothing found (mock all three locations)
   - Test: error message includes all searched paths
   - Use temporary directories or mocks for file existence checks

6. **src/bridge/executor.ts**: Low-level gsd-tools.cjs invocation.
   - `executeGsdCommand(toolsPath: string, command: string, args: string[], cwd: string, timeoutMs?: number): Promise<unknown>`
   - Invokes via `execa('node', [toolsPath, command, ...args], { cwd, timeout: timeoutMs })`
   - Parses stdout as JSON -- throws GsdParseError if JSON.parse fails
   - Maps execa errors: timedOut -> GsdTimeoutError, ENOENT -> GsdToolsNotFoundError, other -> GsdBridgeError
   - Logs command invocation at debug level via shared logger

7. **src/bridge/executor.test.ts**: Unit tests for executor (mocked execa where appropriate).
   - Test: successful command returns parsed JSON
   - Test: timeout produces GsdTimeoutError
   - Test: ENOENT produces GsdToolsNotFoundError
   - Test: non-JSON stdout produces GsdParseError
   - Test: other errors produce GsdBridgeError
