  <name>Task 2: Implement signal parser (extraction from text) and formatter (signal to text)</name>
  <files>src/signals/parser.ts, src/signals/parser.test.ts, src/signals/index.ts</files>
  <behavior>
    - parseSignal extracts signal from text with leading/trailing natural language
    - parseSignal extracts signal from text with ONLY the signal block (no surrounding text)
    - parseSignal returns null for text with no --- delimiters
    - parseSignal returns null for text with --- delimiters but no GSD_SIGNAL marker
    - parseSignal returns null for text with GSD_SIGNAL but invalid/unknown type
    - parseSignal returns null for text with valid type but missing required fields (logs warning)
    - parseSignal handles the first signal block if multiple --- blocks exist (non-greedy)
    - parseSignal treats GSD_SIGNAL:TYPE as a marker line, strips it before YAML parsing
    - formatSignal produces a --- delimited block with GSD_SIGNAL:TYPE on first line, followed by YAML
    - formatSignal output can be parsed by parseSignal (round-trip)
  </behavior>
  <action>
1. **src/signals/parser.ts**: Signal extraction and formatting.
   - `parseSignal(text: string): GsdSignal | null`
     - Use a regex to find `---` delimited blocks: non-greedy match `---\n([\s\S]*?)---` anchored to line starts
     - For each match, check if first line starts with `GSD_SIGNAL:`
     - Extract signal type from the marker line via string split or regex
     - Strip the `GSD_SIGNAL:<TYPE>` line from the block content
     - Parse remaining content with `js-yaml` `yaml.load()`
     - Validate against the type-specific schema from `signalSchemas`
     - If validation fails, log warning with shared logger and return null
     - If validation passes, return the parsed and validated signal object
     - IMPORTANT: Use non-greedy regex (`[\s\S]*?`) to avoid matching across multiple `---` blocks (per research pitfall #4)
   - `formatSignal(signal: GsdSignal): string`
     - Produces a string in the format:
       ```
       ---
       GSD_SIGNAL:<TYPE>
       <yaml-serialized fields minus the 'type' field>
       ---
       ```
     - Uses `js-yaml` `yaml.dump()` for YAML serialization
     - Strips the `type` field from the object before YAML dump (it goes on the marker line)

2. **src/signals/parser.test.ts**: Parser unit tests.
   - Test: signal block surrounded by natural language text -> correctly extracted and parsed
   - Test: signal block with no surrounding text -> correctly parsed
   - Test: text with no `---` delimiters -> returns null
   - Test: `---` block without GSD_SIGNAL marker -> returns null
   - Test: GSD_SIGNAL with unknown type (e.g., GSD_SIGNAL:UNKNOWN) -> returns null
   - Test: valid type but missing required field -> returns null
   - Test: multiple `---` blocks in text -> extracts the one containing GSD_SIGNAL (non-greedy, correct block)
   - Test: round-trip -- formatSignal then parseSignal returns equivalent signal
   - Test: signal with optional fields present -> all fields preserved
   - Test: signal with optional fields absent -> defaults correctly
   - Test each of the major signal categories at least once via parser (not just schemas): PROJECT_READY, DISCUSS_COMPLETE, VERIFY_FAILED, DECISION_NEEDED, AGENT_ERROR
